步骤
🌐 浏览器 / OA
🍃 Spring Boot

1
用户点击
IAM 磁贴
两条路径共用
GET /sso/launch
→

oa.html 里磁贴 href 是 /sso/launch（不再是写死的 OAuth2 端点）。
浏览器发出最普通的 GET 请求，携带当前域名下所有 Cookie。

GET http://localhost:8000/sso/launch
Host: localhost:8000
Cookie: JSESSIONID=ABC123...  ← 如果有 session 才存在
Referer: http://localhost:8000/oa.html

Spring Security Filter Chain 先接手：

// SecurityConfig 里
.requestMatchers("/sso/launch").permitAll()
// → 不拦截，直接放行到 Controller

请求到达 SsoController.launch()，Spring 尝试从当前 Session 里
解析 OidcUser，注入到 @AuthenticationPrincipal。

—
2A
已有 session
直接跳 Dashboard
⚡ 快速路径
← 302 Location: /dashboard.html

浏览器收到 302，立刻跳到 /dashboard.html，用户毫无感知。

← 服务端响应
HTTP/1.1 302 Found
Location: http://localhost:8000/dashboard.html
Set-Cookie: （不变，session 继续复用）
✅ 流程到此结束。全程只有 1 次 HTTP，无需任何 Keycloak 交互。
// SsoController.java
if (principal != null) {
  response.sendRedirect("/dashboard.html");
}
// principal 由 Spring Security 从 Session
// 里的 SecurityContext 里取出并注入

OidcUser principal 不为 null → session 里有有效认证信息。

—（不参与）
2B
未登录
发起 OAuth2
🔄 完整路径
← 302 Location: /oauth2/authorization/keycloak

浏览器紧接着又发一个 GET 请求到 Spring Boot 的 OAuth2 授权端点。

← 服务端响应
HTTP/1.1 302 Found
Location: http://localhost:8000/oauth2/authorization/keycloak
// SsoController.java
if (principal == null) {
  response.sendRedirect(
    "/oauth2/authorization/keycloak");
}
// principal 为 null → 没有 session 或 session 过期
—
3
Spring 生成
授权请求
🔄 完整路径
GET /oauth2/authorization/keycloak →

浏览器请求 Spring Boot 内置的授权发起端点（不是 Keycloak，还在 Spring 这边）。

OAuth2AuthorizationRequestRedirectFilter 处理此请求：

// 1. 生成随机 state（防 CSRF）
state = randomBase64(32)

// 2. 生成随机 nonce（防重放）
nonce = randomBase64(32)

// 3. 把 state 存入 session，供回调时验证
session.setAttribute(
  "SPRING_SECURITY_SAVED_REQUEST", state)

// 4. 构造授权 URL，302 重定向到 Keycloak
← 302 → Keycloak 授权端点（见下方参数）

即将跳来。

4
浏览器 →
Keycloak 登录
🔄 完整路径
GET Keycloak 授权端点 →

浏览器跳到 Keycloak，携带以下所有参数：

GET /realms/{realm}/protocol/openid-connect/auth
  ?response_type=code
  &client_id=iam-client
  &redirect_uri=http%3A%2F%2Flocalhost%3A8000
    %2Flogin%2Foauth2%2Fcode%2Fkeycloak
  &scope=openid%20profile%20email
  &state=7fK2mN...（随机，防 CSRF）
  &nonce=xPq9vR...（随机，防重放攻击）
state：回调时 Keycloak 原样返回，Spring 与 session 里存的值对比，不一致直接拒绝（防 CSRF 攻击）。
nonce：写进 id_token，Spring 验证时对比，防止 token 被重放。
—

Keycloak 检查当前浏览器是否已有 Keycloak SSO Cookie：

场景 A：有 KC_RESTART / KEYCLOAK_SESSION
→ 无需登录，直接静默回调

场景 B：没有 SSO Cookie
→ 显示 Keycloak 登录页面
→ 用户输入账号密码
→ Keycloak 验证通过
两种场景最终都会走步骤 5，区别只是 B 多一个登录页交互。
5
Keycloak →
授权码回调
🔄 完整路径
← 302 Keycloak 回调到 Spring Boot 
GET /login/oauth2/code/keycloak
  ?code=ey...（授权码，一次性，10 秒内有效）
  &state=7fK2mN...（与步骤 3 的 state 相同）
⚠️ code 只能用一次、只在服务端用，绝不能暴露给前端或 URL 历史记录留存太久。

OAuth2LoginAuthenticationFilter 接管：

// 1. 验证 state：与 session 里存的值对比
assert request.state == session.savedState  ✅

// 2. 用 code 换 token（服务端到服务端）
POST Keycloak /token  ← 见步骤 6

Keycloak 发出 302 带 code 和 state。

6
Token 换取
（服务端到服务端）
🔄 完整路径
—（浏览器不参与）
POST Keycloak Token 端点 → 
POST /realms/{realm}/protocol/openid-connect/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=ey...（步骤 5 的授权码）
&redirect_uri=http://localhost:8000
  /login/oauth2/code/keycloak
&client_id=iam-client
&client_secret=（application.yml 里配的）
这是服务端到服务端的请求，client_secret 不会经过浏览器，这也是授权码流程比 implicit flow 安全的核心原因。

Keycloak 验证 code + client_secret，返回三个 token：

HTTP/1.1 200 OK
{
  "access_token": "eyJ...",   ← 访问资源用
  "id_token": "eyJ...",      ← 身份信息 JWT
  "refresh_token": "eyJ...", ← 续期用
  "expires_in": 300,         ← 5分钟
  "token_type": "Bearer"
}
7
解析 id_token
建立 Session
🔄 完整路径
← 302 Location: /dashboard.html

浏览器收到新 Session Cookie，跳到仪表盘。

HTTP/1.1 302 Found
Location: http://localhost:8000/dashboard.html
Set-Cookie: JSESSIONID=XYZ789...
  Path=/; HttpOnly; SameSite=Lax
✅ Session 建立，后续请求只需带 Cookie，不再走 OAuth2 流程。

Spring Security 解析 id_token 的 JWT Payload：

// id_token payload 示例（Base64 解码后）
{
  "sub":   "u001",
  "email": "alice@example.com",
  "name":  "Alice",
  "nonce": "xPq9vR...",   ← 与步骤 3 对比
  "iss":   "http://keycloak/realms/demo",
  "aud":   "iam-client",
  "exp":   1714000000
}

// 构造 OidcUser，存入 SecurityContext
// SecurityContext → HttpSession → JSESSIONID Cookie
—（Token 已发完）
8
Dashboard
加载用户信息
两条路径共用
GET /api/me →

dashboard.html 的脚本调用 fetchMe()，Spring Boot 从 Session 里取 OidcUser 返回用户信息。

← 响应
{
  "authenticated": true,
  "name":          "Alice",
  "email":         "alice@example.com",
  "claims":        { ...完整 OIDC claims... }
}
// UserController.java
@GetMapping("/me")
public ResponseEntity<?> me(
    @AuthenticationPrincipal OidcUser u) {
  if (u == null)
    return ResponseEntity.status(401)...;
  return ResponseEntity.ok(Map.of(
    "name",  u.getFullName(),
    "email", u.getEmail(),
    "claims",u.getClaims()
  ));
}

▶ 完整 HTTP 请求清单

全程 HTTP 交互汇总
⚡ 快速路径（2 个）
GET /sso/launch → 302 /dashboard.html
GET /api/me → 200 用户信息
🔄 完整路径（8 个）
  1. GET /sso/launch → 302 /oauth2/authorization/keycloak
  2. GET /oauth2/authorization/keycloak → 302 Keycloak 授权端点（带 state/nonce）
  3. GET Keycloak /auth?code=...&state=... → Keycloak 登录页 or 静默通过
  4. POST 账号密码（仅首次，有 SSO session 时跳过）
  5. Keycloak → 302 /login/oauth2/code/keycloak?code=...&state=...
  6. Spring Boot → POST Keycloak /token（code 换 token，服务端发）
  7. Keycloak → 200 access_token + id_token + refresh_token
  8. Spring Boot → 302 /dashboard.html（Set-Cookie: JSESSIONID）
关键安全机制
参数 作用 生命周期
state防 CSRF，回调时验证请求期间，验完即删
nonce防 id_token 重放写进 JWT，验完即删
code一次性授权码，服务端换 token10 秒，单次
client_secret换 token 时鉴权永远不经浏览器
JSESSIONID会话标识，绑定 SecurityContext默认 30 分钟
为什么用 /sso/launch
如果 OA 磁贴直接写 /oauth2/authorization/keycloak， 即便已登录也会重走一遍 OAuth2 流程（虽然 Keycloak 有 SSO session 会秒过， 但仍然产生无意义的 4–5 次 HTTP 往返）。
/sso/launch 在 Spring Boot 侧先判断 session，已登录就 1 跳直达，更干净。
